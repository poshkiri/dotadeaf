# InterestingDeaf - Core Rules

## Purpose
This document defines mandatory execution rules for all implementation work in the InterestingDeaf project.
These rules are non-optional and apply to every task.

## Project Context
- Product: platform for deaf and hard-of-hearing Dota 2 players.
- MVP scope: user profiles, player search, chat, and Russian-language Dota 2 patch pages.
- Technology stack: Next.js App Router, React, TypeScript, Tailwind CSS, shadcn/ui, Framer Motion, Supabase, Vercel.

## Mandatory Rules
1. **Read Rule Files First**  
   Cursor must read all files in `/project-rules/` before implementing any task.

2. **No Unapproved Scope Expansion**  
   Cursor must not expand scope independently. Work must stay strictly within the requested task boundaries.

3. **No Invented Features**  
   Cursor must not add, imply, or implement features that were not explicitly requested.

4. **MVP-First Simplicity**  
   Cursor must choose the simplest solution that satisfies the MVP requirements and current acceptance criteria.

5. **Architecture Consistency**  
   Cursor must preserve existing architectural patterns, conventions, and layering decisions across the codebase.

6. **No Generic AI-Looking Design by Default**  
   Cursor must avoid generic, template-like AI visual patterns and should follow project-specific UI/UX direction.

7. **Quality Priorities**  
   Cursor must prioritize clarity, maintainability, accessibility, and security in all implementation decisions.

8. **Implementation Report Required**  
   Cursor must provide a short implementation report after each completed task, including changed files and key outcomes.

9. **Diagnostics Required**  
   Cursor must run relevant diagnostics/checks after each meaningful change and report results.

10. **Environment and Secrets Protection**  
    Cursor must not edit environment files, secret values, or credential-related configuration unless explicitly instructed.

## Enforcement
- If a request conflicts with these rules, Cursor must follow these core rules and ask for clarification before proceeding.
- If required context is missing, Cursor must ask targeted clarifying questions instead of making assumptions.
